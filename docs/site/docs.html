Command Bar (the gdb-style REPL)
CommandDoes
next / nadvance one span
prev / pstep back one span
step / sstep into child spans
finish / fstep out to parent
continue / crun to next breakpoint
whereprint trace position (backtrace)
print <expr>eval against current span, e.g. print messages[-1].content_inline
break <predicate>set a breakpoint; break list to list
delete <id>delete a breakpoint
branch [span_id]branch + replay from a span
diff [a] [b]diff two traces (or a branch vs its parent)
export [trace_id]download a .steth bundle

Keys n p s f c work whenever focus is outside a text field (Esc blurs the prompt).

Breakpoint predicate DSL
break kind == 'tool_call' AND tool_name == 'web_search' AND duration_ms > 5000
break status == 'error'
break NOT (provider == 'anthropic') OR cost_usd > 0.05
break name contains 'planner'

Fields: kind name status tool_name model provider duration_ms tokens_in tokens_out cost_usd error_message. Operators: == != > < >= <= contains, combined with AND OR NOT and parentheses. Hits are detected live during ingestion; the status bar shows ⚑ breakpoint <name> and the matching span is focused.

The .steth bundle
A portable JSON document: trace + all spans, messages, tool_calls, and the llm_cache needed to replay it. Attach it to an incident postmortem; open it anywhere.
Architecture
Python SDK (one-line attach) → OTLP/gRPC → ingestion → DuckDB + Parquet → the Workbench. Replay pins LLM calls via a structural prompt-hash cache (PRD §7.3) and snapshots tool I/O; a branch mutates one input and re-runs deterministically. Full write-up of the deterministic-replay trick: docs/replay.md.
