Stethoscope

"Agents fail silently. We make them speak."

A time-travel debugger for LangGraph agents. The kind of project that exists because the field is so new that nobody had built the gdb-equivalent yet. This document walks the entire codebase, the architecture decisions, every commit on the way, the production AWS deployment, and the parts that don't work yet.

1. The Premise

LLM agents fail silently. You wire up LangChain or LangGraph, give it tools, hand it a prompt, and it does something. When the something is wrong, you read OpenTelemetry spans in Jaeger or print statements in your terminal and squint at a wall of text. There is no equivalent of rewind and try this with one input changed, which is the central move of every debugger that came before.

Stethoscope is what happens when you build that equivalent. It records every span the agent emits, stores them on disk, gives you a tree-shaped UI to step through, and lets you branch from any moment with a mutated input to see what would have happened. Then it lets you diff the two runs.

That sentence is the whole product. The rest is making it actually work.

2. The Three Pillars

OBSERVE

See exactly what happened, at any granularity. Span tree, prompts, completions, tool inputs, tool outputs, token counts, cost. Everything the agent did, laid out so your eye can scan it.

REPLAY

Re-run any moment with modifications. Pin every LLM response to what was recorded, override the one tool result you care about, and rerun the agent. New trace shows up as a fork of the original.

COMPARE

Diff two runs to understand divergence. The alignment shows which spans match, which diverged, and which span is the actual cause of the different outcome.

3. The Data Model

Two ideas carry the whole system. A trace is one run of an agent from start to finish. A span is one operation inside that run. You have spans for LLM calls, tool calls, node executions, anything the agent did that you want to see.

Spans nest. The agent's root span contains the planner's span, which contains the LLM call span. The trace tree in the UI walks parent and child relationships among spans, sorted by time. Each span carries timestamps, status, costs, tokens in and out, and the attributes the SDK attached.

This is standard OpenTelemetry. The reason that matters is interoperability. Anyone instrumenting an agent with OTel can ship traces to Stethoscope. The SDK just gives them nicer convenience wrappers around the OTel API.

4. Local Capture Flow

Three pieces work together. The Python SDK is what an agent author imports. They call stethoscope.attach(graph) and the SDK wires OpenTelemetry into their LangGraph instance. Every node execution, every LLM call, every tool call becomes a span. The SDK ships those spans over OTLP/gRPC to wherever you've pointed it, default localhost:4317.

The store at ~/.stethoscope/projects/<name>/traces.db is DuckDB chosen because it is an embedded analytics database, fast for the kind of give me all spans for this trace ordered by time queries the UI does, and you do not have to run a separate server. One file per project keeps things simple.

So when your agent runs, traces flow through the wire into a file on disk in about fifty milliseconds. That is the OBSERVE pillar working.

Try it locally (3 commands) 
# Terminal 1: install SDK + start the ingestion server
pip install -e packages/sdk-python
python -m tools.ref_ingest serve

# Terminal 2: run the example agent
python examples/replayable_agent/agent.py

# Terminal 3: check it landed
curl http://127.0.0.1:4318/traces

5. The Workbench UI

packages/ui is a React app that talks to the read API the ingestion server exposes on 4318. It has three panes. Left is a list of traces. Middle is the span tree for whichever trace you selected. Right is the inspector, which shows everything about whichever span you clicked.

The UI is deliberately ugly. Hand-rolled CSS, IBM Plex Mono, Press Start 2P for headers. The README's "an interface that feels like debugging C in 1998" line is the aesthetic. The reason is that LLM tooling in 2026 is almost all chat-shaped, which is the wrong shape for debugging. You want dense information laid out so your eye can scan it.

6. Replay, The Actual Point

The killer feature is branching. You see a span where the agent did something dumb. You think, "what if that tool had returned the right thing instead." You click branch, edit the tool result, and the system replays the agent from that span with everything else pinned.

The replay engine (tools/ref_replay) builds a manifest. That manifest says for this span the LLM should return this exact completion, for this tool span the tool result is this overridden value. It runs the agent's entrypoint as a subprocess with the manifest in an environment variable. The agent code in examples/replayable_agent/agent.py checks for that env var and uses the recorded responses instead of calling the real LLM or tool. New spans flow back through OTLP, get stored, and show up in the UI as a fork of the original trace.

For this to work, the agent has to be deterministic given the same inputs. That is why the example agent uses a fake LLM and a fake tool. Real agents are mostly deterministic once you pin the LLM responses, which Stethoscope does by storing every prompt's hash and response. PRD section 7.3 explains the cache. Section 4.3 explains why structural hashing beats byte-level hashing in practice.

7. Diff

If you have two traces, you can diff them. crates/diff and its Python mirror align the two span trees, mark which spans are identical, which diverged, and where exactly the agent took a different branch. The DiffView pane shows them side by side with color coding.

In the killer demo, you diff the original failed order-bot run against the replayed successful one. The visualization shows the inventory_lookup span as the only changed input, and everything downstream as a consequence of that change. That is a clean one variable changed, here is the blast radius story which is exactly what scientific debugging is.

8. Rust Spec and Python Reference

The architecture is intentionally double-implemented. The Rust crates are the canonical spec. The Python tools are a runnable reference. Same schemas, same mapper logic, same store semantics, same wire format.

The reason is that the project started on a machine without admin rights to install a Rust toolchain. The Python reference let you ship and verify the whole vertical slice in one language while leaving the Rust crates as the production target. As of v3.x the Rust workspace builds and tests cleanly on Linux and macOS through CI, with documented Windows quirks.

The cargo workspace excludes the Tauri shell because pulling in WebView2 makes the test build heavy. When you ever want to ship a real Stethoscope desktop binary, the Tauri shell calls into the Rust crates over IPC and you get one signed executable per OS. The Python tools become test fixtures and developer ergonomics.

9. Cloud Architecture

cloud/api is the same logic running as a multi-tenant FastAPI service. Every row in Postgres carries a tenant_id. The API key in the X-Stethoscope-Key header maps to a tenant. An agent in any customer's environment can ship traces to your hosted endpoint and only see its own traces back. Cognito handles human user auth for the Workbench login. JWT tokens for sessions, long-lived API keys for programmatic agent traffic.

The cloud architecture put up on AWS is exactly what the Mermaid diagram in the README shows. CloudFront fronts everything for free TLS. ALB lives behind it with a secret-header check so direct ALB traffic gets 403'd. ECS Fargate runs the FastAPI app and, separately, the replay worker. RDS Postgres sits in private subnets. SQS holds replay jobs. S3 carries large payloads. Cognito stores users. CloudWatch alarms and AWS Budgets so you don't accidentally bankrupt yourself.

The honest gap. The cloud replay worker cannot actually run a customer's agent because the customer's source code is not on your server. There is a stub that logs the request and returns 501, plus the SQS queue and worker task definition so the shape is right for a portfolio review. The workaround for a real Stethoscope-as-a-service would be a customer-side runner that subscribes to the replay queue. Documented in the RUNBOOK.

11. The Grim Part

Everything that does not make the screenshot.

Norton 360 vs cargo target

Norton 360 treats Rust's target/ directory as a malware factory. It holds files open while it scans them. cc-rs needs to write to those files at the same time. You get link errors that look like the linker cannot find a library, when actually the linker cannot open a file the antivirus is reading. The workaround is to add Norton exclusions for target/ and ~/.cargo/. The real fix is to leave Windows entirely and build in WSL.

libduckdb-sys on MSVC without /EHsc

DuckDB's Rust bindings on Windows MSVC don't pass /EHsc to the C++ compiler. Half the standard library uses exceptions. The build dies on an unrelated-looking link error about duckdb.lib not existing. The actual fix is one line in .cargo/config.toml setting CXXFLAGS_x86_64-pc-windows-msvc=/EHsc. Finding it took ninety minutes.

AWS retired Postgres 16.4

AWS retired Postgres 16.4 in ap-south-1 two weeks before this project tried to use it. Pin to the major version and AWS picks the latest patch. Five minutes of debugging for a Terraform edit.

Em dashes in Security Group descriptions

You cannot use an em dash in an AWS Security Group description. The error message is "Character sets beyond ASCII are not supported." Learned this from inside a thirty-second window where the NAT gateway was the only thing running, ticking at ₹3 per hour.

ECS lifecycle ignore_changes vs first deploy

The ECS service has lifecycle { ignore_changes = [task_definition, desired_count] } so GitHub Actions deploys do not get rolled back by a subsequent terraform apply. But on a fresh first deploy, the service stays pointed at the bootstrap task definition revision while terraform creates a new revision with the real image tag. Symptom: CannotPullContainerError: stethoscope-api:bootstrap not found. Fix: manually aws ecs update-service --task-definition $LATEST --force-new-deployment, which is exactly what the deploy workflow would have done.

AAS tag values cannot be empty

The provider default_tags in versions.tf set Repo = var.gh_repo. When you leave gh_repo empty in tfvars, every other resource accepts an empty tag, but Application Auto Scaling refuses. Fix: merge({...}, var.gh_repo == "" ? {} : { Repo = var.gh_repo }).

PowerShell pipe ECR login

PowerShell's pipe encodes the password with an extra newline or BOM that Docker's stdin reader chokes on. aws ecr get-login-password ... | docker login --password-stdin ... fails with HTTP 400 from ECR. Workaround: store the token in a variable and pass via --password directly, or use cmd /c to bypass PowerShell's pipe.

The apply-destroy cycle

Production-shaped portfolio work lives or dies on the apply-destroy cycle. Anyone can draw boxes. The skill people actually care about is whether you remembered to terraform destroy before bed.

CloudFront takes three minutes to provision and another three to delete. RDS takes seven. Most of an AWS Saturday is waiting.

13. What's Left

  • Replay in cloud needs a customer-side runner story. The worker is a stub. Real implementation needs a per-tenant CLI that subscribes to SQS, runs the agent locally, and ships traces back.
  • The Tauri desktop shell compiles but has not been launched with a real pnpm tauri dev end to end. WebView2 on Windows and webkit2gtk on Linux both work for cargo check; full GUI run is the next step.
  • The Workbench is functional but spartan. The state watcher and live token stream panes are wired but not richly populated yet.
  • The replay engine handles tool_response mutations. The other mutation types in PRD section 5 (llm_message, state_patch, condition) are spec'd but not wired in the reference.
  • There is no paying-customer auth flow. Cognito user pool exists, signup works, but there is no billing or plan tier.
  • The Windows Rust test failure in stethoscope-diff on the CI Windows matrix entry is almost certainly a CRLF vs LF issue in an expected-output literal. Trivial fix once you grab the CI log.

Everything else works.
